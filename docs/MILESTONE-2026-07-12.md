## Acceptance decision

DwellProof is accepted as a controlled local prototype and evidence/calculation
baseline. It is **not** accepted as an actionable property-investment advice
system. The current UI and deterministic core must refuse a directional result
until one real property case supplies all required evidence.

## Safety invariants

- Missing, stale, conflicted, future, cross-scope or wrong-role evidence fails
  closed; readiness never becomes a BUY/SELL/HOLD label.
- Seven unique, verified and clear due-diligence checks from one identified
  case, property and borrower are required before analysis.
- City price indices are market context only; they are not property valuations
  or comparable transactions.
- LPR is benchmark context only; loan calculations require a dated verified bank
  execution quote and revalidate every schedule row.
- Standard scenario results retain complete provenance and reject one evidence
  ID mapping to different facts.
- The `legacy/` tree remains `LEGACY_ARCHIVE_UNVERIFIED`, non-actionable, and
  visibly separate from DwellProof.
- The frontend is a locked presentation shell until a tested API or signed
  calculation snapshot connects it to `core/`.

## Remaining boundary

A trustworthy property case still requires recent attributable comparable
transactions, authoritative title/disposal/encumbrance checks, versioned local
taxes and restrictions, a written bank execution-rate quote, complete holding
and exit costs, source-freshness handling, and end-to-end tests. A public input
boundary must also impose a documented maximum holding-period/resource limit.

The next stage is complete only when one real property can travel from evidence
capture through a reproducible signed calculation snapshot and UI rendering.
Even then, the system must not automatically emit a directional investment
recommendation without a separately specified and validated decision policy.
